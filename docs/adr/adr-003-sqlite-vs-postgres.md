# ADR-003: SQLite 而非 PostgreSQL 作为数据库

## 状态

✅ 已采纳 (2026-05-27)

## 背景

选择面试助手的后端数据库。

## 方案对比

| 维度 | SQLite | PostgreSQL |
|------|--------|-----------|
| 部署成本 | 零配置 | 需独立服务 + 运维 |
| 备份 | `sqlite3 .backup` | pg_dump |
| 并发 | WAL 模式可处理少量并发 | 高并发 |
| 性能（单机） | 读 ~120k QPS | 读 ~80k QPS |
| 复杂查询 | 基础 SQL | 全文索引、窗口函数等 |

## 决策

选 **SQLite**。

## 理由

1. **单用户场景** — 绝大多数操作只有一个用户，不需要多写并发
2. **零运维** — 不需要安装/配置/守护进程，文件即数据库
3. **备份简单** — 一个 `sqlite3 .backup` 命令即可在线热备
4. **索引足够** — LIKE + 索引可满足面试历史搜索需求

## 后果

- 多写并发瓶颈，但面试场景是"读多写少"（一次答题写入，多次查询报告）
- Docker 部署时需将数据库文件映射到持久卷
- 数据量达到 ~5GB 后需考虑迁移
